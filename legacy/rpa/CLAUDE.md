# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

This repo is a single-file HTML/JS prototype (`j.html`) of a corporate expense-approval ("지출품의") system with an embedded AI chatbot agent. There is no build system, package manager, or test framework — it is a static file meant to be opened directly in a browser.

`app.py` is a Streamlit port of the same system (`pip install -r requirements.txt && streamlit run app.py`) that mirrors `j.html`'s business logic function-for-function; see its own module docstring for where the two intentionally diverge (chat panel fixed to the center of the main area instead of a floating widget, no blocking `alert()`/`confirm()`, an extra `SHORTFALL_CONFIRMATION` chat step). `app.py`'s chatbot has continued to diverge from `j.html` since the initial port — see "app.py chatbot state machine" below before assuming the two are still in lockstep.

`app.py`'s screen layout is also inverted from what you'd expect: the **ERP tabs (지출결의입력/승인/예산내역) live in the sidebar**, and the **AI chatbot is centered in the main area** (via `st.columns([1, 5, 1])`) — Streamlit's built-in sidebar collapse arrow doubles as the "show/hide ERP screen" toggle, so there's no custom show/hide button to maintain.

`requirements.txt` pins `streamlit==1.49.0` exactly (not `>=`) — see "Streamlit gotchas" below for why the exact floor matters.

`PRD.md` is the **source of truth for business rules**. It was written by reverse-documenting the behavior actually implemented in `j.html`. When the two disagree, treat it as a bug to flag to the user rather than silently "fixing" one side — ask which one is correct before changing anything (this happened once already: PRD originally said both 회의비 and 조직활성화비 require PDF proof over ₩100,000, but the code only enforces it for 회의비; PRD.md §3.3 was corrected to match the code).

Any change to business logic in `j.html` (budget rules, date rules, chatbot parsing/templates, etc.) should be reflected back into the matching section of `PRD.md`, and vice versa — keep them in sync.

## Running / testing

There is no CLI build or test step. To "run" the app:

```bash
open j.html   # macOS, opens in default browser
```

Verify changes by opening the file in a browser and exercising the UI directly (there is no automated test suite). Key manual flows to check after any change:
- Tab 1 (지출결의 입력): select dept → select card → check usage rows → confirm auto-recommended account/amount/PDF-requirement behavior
- Tab 4 (지출품의 승인): approve and cancel(반려) flows, including multi-line shortfall-split cancellation restoring the card usage row
- Chatbot: 💳 virtual-usage button, free-text "날짜 시간 상호 금액 품의해줘" command, account-change command, budget-query command, confirmation flow
- Chatbot approval/cancel: ID-based (`"1001번 승인해줘"`), condition-based (dept/amount/content, e.g. `"IT정보팀 미승인 건 취소해줘"`), and bulk (`"전체"`/`"모두"`/`"전부"`/`"일괄"`) commands for both 승인 and 취소(반려) — check the ambiguous-candidate-list path (condition search, no explicit ID/bulk, >1 match) and that approved (non-미승인) items are correctly excluded/reported as unavailable
- `app.py`-only additions (not in `j.html`): the "💰 예산조회"/"📝 품의"/"✅ 승인" quick-action buttons (each just sends a canonical phrase through the normal NLP pipeline — verify they produce the same result as typing the phrase); the no-filter card-command path, which now lists candidates for the user to pick by number (`AWAITING_CARD_SELECTION`) instead of picking one at random; the dedicated 참석자/사유 `st.form` inputs; the ✅/❌ confirmation buttons on all three confirmation-type steps; and the chatbot-side 회의록 PDF upload gate (10만원 이상 본사회의비 건은 `CONFIRMATION` 단계에서 파일 없이 긍정 응답해도 상신되지 않는지, 새 품의 흐름을 시작하면 이전 건의 첨부파일이 초기화되는지 확인)

## Architecture (single file, three `<script>` blocks)

`j.html` has no modules/build step — everything is global functions/state inside plain `<script>` tags, in this order:

1. **Chatbot visibility toggle** — trivial show/hide of `#chatbot-container` / `#chatbot-launcher`.
2. **Core ERP app** (data, tabs, forms) — owns the canonical state and is what the chatbot script reuses:
   - `BUDGET_CODES`: the 5 fixed 적요/예산코드 (08WA/40AA/45BC/12CC/20DD) with `allocated` amounts.
   - `dataset`: array of 품의 line items (`id`, `dept`, `date`, `title`, `amount`, `account`, `status`, `sourceUsageKeys`), persisted to `localStorage` via `loadDataset()`/`saveDataset()`. `renderAll()` re-renders the approval table, status report table, and budget table, and always calls `saveDataset()` first — so any code path that mutates `dataset` must end by calling `renderAll()` (or call `saveDataset()` directly if it skips a full render).
   - `DEPT_CARDS`: static map of department → corporate cards.
   - `cardUsageCache` / `randomUsageForCard()`: lazily-generated random card transactions per card number, cached in `localStorage` (`expenseDemoCardUsageCache`) so re-selecting a card returns the same rows.
   - `usedCardUsageKeys` (`localStorage` key `expenseDemoUsedCardUsage`): a card-transaction is "used" once it's been submitted into `dataset`; used keys are filtered out of both the manual card-usage table (`renderCardUsageTable`) and the chatbot's candidate pool (`collectDeptCardUsage`). `cardUsageKey(cardNumber, item)` is the composite identity for a transaction (no stable id exists otherwise).
   - `buildLedgerEntries(accountValue, amount)`: the shortfall-split budget logic — if a 회의비/조직활성화비 request exceeds its own remaining budget, the shortfall is split into a second entry on 08WA (본사 업무추진비); 08WA itself never splits further (hard block via `alert`). Returns `null` (after alerting/confirming) when submission should be aborted — every caller must check for `null` and stop.
   - `validateDocDate()` / `ENFORCE_DOC_DATE_RULE`: 월/수/금-only + no-23rd rule, gated by the `ENFORCE_DOC_DATE_RULE` flag (currently `false`, i.e. test preset — do not flip to `true` unless asked, per the inline `// TODO` at that line).
   - `submitForm()` / `processApproval()` / `cancelApproval()`: manual-form submission, approval, and approve-cancel(반려). `cancelApproval()`'s restore rule is the trickiest part — a card-usage key is only unmarked as "used" if *no remaining* `dataset` line still references it via `sourceUsageKeys` (handles multi-line shortfall-split cancellation correctly).
3. **Chatbot NLP + conversation state** — reuses everything above (`BUDGET_CODES`, `dataset`, `DEPT_CARDS`, `randomUsageForCard`, `buildLedgerEntries`, `validateDocDate`, `getBudgetRemains`, etc.) rather than duplicating business logic:
   - `chatbotState.step`: a small state machine with two independent paths out of `IDLE` — `IDLE → AWAITING_ATTENDEES → CONFIRMATION → IDLE` (new document from a card transaction) and `IDLE → APPROVAL_CONFIRMATION → IDLE` (승인/취소 command). `sendChatMessage()` dispatches on this plus a couple of always-on intents (`isBudgetQuery`, `tryHandleAccountChange`) that short-circuit the step machine regardless of current step. `chatbotState.pendingTransaction`/`recommendedProposal` back the first path; `chatbotState.pendingApprovalAction` (`{ type: 'approve'|'cancel', ids }`) backs the second — kept as separate fields so the two flows can't clobber each other's state.
   - `parseTransactionQuery()` / `storeMatches()` (Levenshtein-based fuzzy match) / `findCardTransactionByQuery()`: turn free text into a transaction match against `collectDeptCardUsage()`'s candidate pool, returning either a single match, an `ambiguous` list (never auto-pick when >1 match), or `not-found`.
   - `parseAttendeeInput()` / `buildAttendeeLabel()` / `buildPurposeForAccount()`: turn "참석자 / 내용" free text into the auto-generated 사유 문구, templated per account (회의비/조직활성화비/업무추진비 each have a distinct template — see PRD.md §5.2C). When `tryHandleAccountChange()` switches the account mid-flow, the purpose string must be rebuilt via `buildPurposeForAccount` with the new account, not just relabeled.
   - Confirmation-step submission mirrors `submitForm()`'s entry-building/validation (same `buildLedgerEntries` + `validateDocDate` calls) rather than reusing `submitForm()` directly, since the chatbot path doesn't go through the DOM form fields — keep both paths' budget/date validation in sync if either changes.
   - `isApprovalCommand()` / `isCancelCommand()` (cancel wins if both match) → `parseApprovalTargetQuery()` extracts `ids`/`dept`/`amount`(+`이상`/`이하`)/`content`/`bulk` from free text → `findApprovalTargets()` filters `dataset` to `status === '미승인'` plus those conditions. `handleApprovalCommand()` only shows the ambiguous-candidate list (mirroring the card-search `ambiguous` case, §5.2F) when there's **no** explicit ID and **no** bulk keyword and still >1 match — an explicit ID list or a `전체`/`모두`/`전부`/`일괄` keyword always proceeds straight to confirmation with however many matches it found. `executeApprovalAction()` re-filters by id + `status === '미승인'` at execution time (not the objects captured earlier) in case the dataset changed while the confirmation was pending, then applies status-flip or delete. **Note:** its cancel branch re-implements `cancelApproval()`'s sibling-line restore rule (only unmark a card-usage key as unused if no remaining `dataset` line still references it) inline rather than calling `cancelApproval()` — if that restore rule ever changes, update both places.

## app.py chatbot state machine

`app.py`'s `chatbot_step` has grown beyond the two paths described above for `j.html`. Current states: `IDLE` → (card command with a filter) → `AWAITING_ATTENDEES` → `CONFIRMATION` → `IDLE`; `IDLE` → (card command with **no** date/time/store/amount filter, e.g. `"품의하겠다"`) → `AWAITING_CARD_SELECTION` (numbered candidate list; reply with a number to proceed) → `AWAITING_ATTENDEES` → ...; `IDLE` → (approve/cancel command) → `APPROVAL_CONFIRMATION` → `IDLE`; and mid-flow, a budget shortfall can insert `SHORTFALL_CONFIRMATION` before the document is actually created. `is_card_command`/`is_approval_command`/`is_cancel_command` are checked in that priority order in `send_chat_message`'s `IDLE` branch — an approval-shaped phrase (`"결재해줘"`) is always caught before the looser card-command noun-gate (`품의`/`결재`/`상신`) gets a chance to misfire.

Extending trigger words: `is_card_command`/`parse_transaction_query` treat `품의`/`품의서`/`결재`/`상신` as interchangeable core nouns, accept a growing list of verb endings (`해줘`/`해주`/`써줘`/`하겠다`/`할래`/`할게`/...), and also fire when the message ends with a bare core noun and no verb at all (e.g. a pasted `날짜 상호 금액 품의` line with no trailing verb). When none of that matches but a core noun is still present, `send_chat_message` asks a clarifying question instead of guessing — don't remove that fallback when adding new verbs, it's the safety net for phrasing nobody anticipated. Any new trigger word must be added to *both* the recognition regex and `parse_transaction_query`'s stripping regex, or the leftover verb text gets misread as the store name.

Three confirmation-type steps (`CONFIRMATION`, `SHORTFALL_CONFIRMATION`, `APPROVAL_CONFIRMATION`) share one rendering block: ✅/❌ buttons that just call `send_chat_message("응")` / `send_chat_message("아니오")`, reusing each step's existing `POSITIVE_RE` branch rather than adding button-specific logic.

## Streamlit gotchas (app.py)

- **Never reassign `st.session_state[key]` for a widget after that widget has already been instantiated with that `key` in the same script run** — Streamlit raises `StreamlitAPIException` and aborts the rest of the run (this silently ate the whole submit handler once, including the `send_chat_message` call after it). For the 참석자/사유 inputs, use `st.form(..., clear_on_submit=True)` + `st.form_submit_button` and read the widgets' local return values instead of touching `st.session_state` post-hoc. The chatbot's `chat_meeting_minutes_file` (`st.file_uploader` key) is reset a different, safe way: `analyze_and_recommend()` sets it to `None` when a *new* proposal starts, which always runs in an earlier script pass than the one where `chatbot_step == "CONFIRMATION"` first becomes true and the uploader widget itself gets instantiated — so the reset and the widget's instantiation never land in the same run.
- The chat log (`st.container(height=460, key="chat_log_box")`) does **not** auto-scroll when new messages are appended. The fix is a `components.html` snippet that reaches through `window.parent.document` (components render in an iframe), finds the **last user message** (via `[data-testid="stChatMessageAvatarUser"]` → closest `[data-testid="stChatMessage"]` ancestor), and sets `chat_log_box`'s own `scrollTop` so that message lands at the top of the box — re-run every script pass, not just once. Two things this deliberately does *not* do:
  - It does **not** scroll to the bottom of the container. Whatever the user just typed/selected is followed by a bot reply and often more UI below it in the same box (the AI recommendation, the 참석자/사유 form, confirmation buttons) — scrolling to the bottom hides that follow-up content, while pinning the user's own message to the *top* of the box shows everything that comes after it, for as much as fits.
  - It does **not** call `.scrollIntoView()` on the target element. `scrollIntoView()` walks up and scrolls *every* scrollable ancestor needed to bring the element into view — including the outer page itself — so it visibly jumps the whole page's scroll position, not just the chat box (this actually happened once). Compute the offset manually instead (`target.getBoundingClientRect().top - box.getBoundingClientRect().top`, added to `box.scrollTop`) so only `chat_log_box`'s internal scroll changes and the page's own scroll position is left untouched.
  
  Keep the `components.html(...)` call **inside** `chat_box`, not after it — even with `height=0` the component's iframe wrapper still reserves visible block-level space in normal document flow, showing up as an empty gap between the chat log and whatever comes next (e.g. `st.chat_input`).

  **Forward-compat risk**: on the Streamlit version actually installed on this machine (1.59.0, well past this repo's `==1.49.0` floor), calling this emits `Please replace st.components.v1.html with st.iframe — st.components.v1.html will be removed after 2026-06-01` — a date that has already passed as of this writing. It still works today, but treat this as due for a rewrite onto `st.iframe` (or whatever replaced it) the next time this file is touched, rather than assuming the deprecation warning is stale/harmless.
- `width='stretch'` (string-based sizing, replacing `use_container_width`) only works correctly on `st.data_editor` from Streamlit **1.49.0** onward — 1.48.0 added the parameter but its `data_editor` implementation still does a raw int-only `proto.width = width` internally and throws `TypeError: 'str' object cannot be interpreted as an integer`. This is why `requirements.txt` pins `streamlit==1.49.0` exactly rather than a loose `>=` range.
- Where a block of `with`-guarded code lands visually (`st.sidebar` vs. `st.columns(...)` vs. plain main area) does **not** change when it executes — that's still purely determined by the code's top-to-bottom position in the script. This is why the ERP tabs (now `with st.sidebar:`) still have to come *before* the chatbot block (now `with chat_col_mid:`, itself further down the script) even though visually the sidebar is off to the side: the chatbot block calls `st.rerun()` in several places, and if it ran first, a rerun would cut the script off before tab1's `dept_code`/`doc_date` widgets ever executed that pass, and Streamlit would garbage-collect their session_state keys (`KeyError` next run). Don't reorder these two blocks without preserving that relationship.

## app.py styling

Light theme + indigo (`#6366F1`) accent is set in `.streamlit/config.toml`; anything config.toml can't express (button radius, card/chat-message shadows, the chatbot status badge/pulsing dot, the gradient "AI 지출결의 에이전트" banner) is injected once as `CUSTOM_CSS` right after `st.set_page_config`. Prefer `st.container(border=True)` for new card-style groupings over hand-rolled HTML — see PRD.md §8 for the full design rationale.

`.streamlit/config.toml` didn't actually exist for a long stretch of this app's history despite PRD.md §8 and `app.py`'s own CUSTOM_CSS comment both describing it as the source of the theme — the app was silently falling back to Streamlit's default (non-indigo) theme the whole time, which is why it looked notably plainer than the CSS/PRD implied. If the app ever looks "off-theme" again, check that this file still exists before assuming the CSS is broken.

`st.chat_input()` only pins to the bottom of the screen when called directly in the bare main container; inside `st.sidebar`, `st.columns(...)`, or any other nested container it renders inline, in normal document flow. That's why the "🔧 테스트 도구" buttons (virtual card usage, reset test data) can simply be placed *after* the centered chat column's `chat_input` call to appear below it — don't assume it always sticks to the bottom like it does in the bare main area.
