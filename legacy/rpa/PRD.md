# PRD.md (Product Requirements Document)

## 1. 프로젝트 개요 (Overview)

본 시스템은 기존 ERP의 지출품의 화면을 현대화하고, AI 챗봇 기능을 결합하여 사용자가 자연어로도 지출품의를 조회·상신할 수 있도록 지원하는 '지출품의 처리 및 AI 챗봇 연동 시스템'이다. 본 문서는 단일 파일 프로토타입(`j.html`)에서 검증된 비즈니스 규칙과 기능을 실제 구현체(`expense-system/`)에 반영하기 위한 요구사항을 정의한다.

## 2. 기본 환경 설정 및 데이터 사양 (System Defaults & Data Specifications)

### 2.1 기본 조직 및 사용자 정보

* **기본 품의부서**: `C30 IT정보팀` (시스템 진입 시 기본 선택 및 표시 부서)
* **대표 카드 소유자**: C30 부서 소속 카드 중 하나인 `4521-8890-1123-7765` 카드의 소유자는 '김민준'으로 지정한다.
* **보안 규정**: UI상에서 법인카드의 비밀번호(**법카비번**) 정보는 일절 노출하지 않는다.

### 2.2 입력 창 UI 정책

* **적요(상세 내역 기재) 입력**: 별도의 placeholder나 고스트 텍스트(입력 시 사라지는 가이드) 없이 **빈 입력창**으로 제공하여 직관성을 높인다.

### 2.3 데이터 지속성 (Persistence)

* 모든 품의/승인 내역 데이터(`dataset`) 및 카드별 랜덤 사용내역 캐시(`cardUsageCache`)는 로컬 스토리지(`localStorage`) 또는 데이터베이스에 실시간 저장되어, **새로고침을 하거나 재진입해도 동일하게 유지**되어야 한다.
* 초기 데이터 세팅 시 샘플(시드) 데이터는 제공하지 않으며, **빈 상태로 시작**한다.
* 개발 및 테스트 편의를 위해 데이터 및 캐시를 일괄 소거하고 서비스를 초기 상태로 되돌리는 **'테스트 데이터 초기화'** 기능을 제공한다.

## 3. 핵심 비즈니스 규칙 (Core Business Rules)

### 3.1 예산코드(적요) 5종 정의

시스템은 다음 5가지 계정과목을 관리하며, 각각 독립된 배정 예산, 사용 예산, 잔여 예산을 가진다.

1. **본사 업무추진비 (08WA)**
2. **본사 회의비 (40AA)**
3. **본사 조직활성화비 (45BC)**
4. **(일반)전화료 (12CC)**
5. **교통비 (20DD)**

### 3.2 품의일자 제한 규칙 (`ENFORCE_DOC_DATE_RULE`)

* **원칙**: 지출품의 상신일(일자)은 **월요일, 수요일, 금요일**만 가능하며, **매월 23일**에는 품의를 올릴 수 없다.
* **시스템 제어 플래그**: 해당 규칙은 `ENFORCE_DOC_DATE_RULE` 플래그(Boolean)를 통해 제어한다.
  * `true`: 규칙 엄격 적용 (실운영 환경)
  * `false`: 규칙 비활성화 (테스트 환경 프리셋)

### 3.3 계정별 자동 판정 및 분기 규칙

법인카드 승인 내역을 기반으로 품의를 작성할 때, 시스템은 다음 규칙을 자동 적용한다.

* **회의비(40AA) 조건**: 카드 사용 항목의 결제 위치가 '여의도'이고 결제 시간이 '20:00 이전'인 경우에만 추천/인정된다.
* **증빙 서류 필수 한도**: **회의비(40AA)** 항목 중 결제 금액이 **100,000원 이상**일 경우에 한해 '회의록 PDF 업로드'를 필수 요구사항으로 강제한다. (조직활성화비는 회의비 요건 미충족 시 대체 배정되는 계정이므로 별도의 PDF 업로드 요건을 두지 않는다.)
* **예산 부족 시 자동 분할 (Shortfall Splitting)**:
  * '회의비' 또는 '조직활성화비'로 판정된 건이 해당 부서의 배정 예산 잔액을 초과할 경우, **초과된 부족분 금액은 '본사 업무추진비(08WA)' 계정으로 자동 분할**되어 다중 라인으로 생성된다.
  * 단, 분할되어 넘어간 '본사 업무추진비'마저 예산이 부족할 경우 예산 초과로 판정하여 **품의 제출을 최종 차단**한다.

## 4. 카드결제 상태 추적 및 승인취소 흐름

### 4.1 "사용됨" 상태 추적 (Lifecycle)

* 지출결의로 상신 처리된 카드결제 내역은 **'사용됨(`usedCardUsageKeys`)'** 상태로 마킹된다.
* 사용됨 상태인 결제 내역은 지출결의 입력 탭의 선택 목록 및 챗봇 검색 대상 후보에서 **자동 제외**된다.
* 모든 품의 항목은 원본 데이터 추적을 위해 매칭되는 카드결제 키(`sourceUsageKeys`)를 반드시 기록 보관해야 한다.

### 4.2 부서장 승인취소(반려) 프로세스

* **'지출품의승인(부서장)'** 탭에서 미승인 건에 대해 '선택 항목 승인취소' 처리가 가능하다.
* 취소 시 해당 품의 내역은 `dataset`에서 제거된다.
* **복원 규칙**: 예산 분할(Shortfall Split) 등으로 인해 하나의 카드 결제 건에서 여러 개의 품의 라인이 파생된 경우, **해당 카드 결제 키를 참조하는 모든 품의 라인이 취소(삭제)되어야만** 카드 내역의 '사용됨' 마킹이 해제되고 다시 선택 가능한 상태로 복원된다.

* **챗봇 연동**: 위 승인/승인취소 처리는 '지출품의승인(부서장)' 탭 UI뿐 아니라 AI 챗봇을 통한 자연어 명령으로도 동일하게 수행할 수 있다. 상세 규칙은 §5.2F 참조.

## 5. AI 챗봇 기능 요구사항 (AI Chatbot Specifications)

### 5.1 챗봇 UI/UX 및 레이아웃

* **화면 배치**: 우측 하단에 고정된 원형 🤖 런처 버튼을 제공하며, 클릭 시 챗봇 창이 토글(`toggleChatbot`)된다. 챗봇 상단에는 숨기기(─) 버튼을 배치한다.
* **UI 스타일**: 가상 카드결제 발생 버튼은 아이콘 형태로 최소화하고, 챗봇 위젯의 너비를 충분히 확보하여 타이틀 문구의 줄바꿈 현상을 방지한다.

### 5.2 자연어 명령 처리 (NLP Parsing)

#### 진입 방식: 빠른 실행 버튼

대화가 `IDLE` 상태(새 요청을 기다리는 상태)일 때, 채팅창에는 자유 텍스트 입력과 별개로 **"💰 예산조회" / "📝 품의" / "✅ 승인"** 3개 버튼을 항상 노출한다. 각 버튼은 아래 절의 자연어 명령과 동일한 대표 문구(`"예산 얼마 남았어?"` / `"품의하겠다"` / `"승인해줘"`)를 보낸 것과 완전히 동일하게 처리되어(§5.2A/D/F의 기존 파싱·후보 목록 로직을 그대로 재사용), 사용자가 정확한 명령 문구를 몰라도 세 가지 핵심 흐름 중 하나로 바로 진입할 수 있게 한다. 버튼은 진입점일 뿐이므로, 이후 세부 조건(날짜/상호/금액, 품의번호, 부서 등)은 여전히 자유 텍스트로 보충 입력한다.

#### A. 카드결제 검색 및 자동 품의

* 사용자가 채팅창에 `"1월 5일 19:15 스타벅스 88000원 품의해줘"` 형태로 명령할 경우, 시스템은 **날짜, 시간, 상호명, 금액** 조건을 파싱한다.
* **핵심 명사 인식**: `품의`(`품의서` 포함) 외에 `결재`, `상신`도 품의 작성 요청의 핵심 명사로 인식한다. (단, `결재해줘`/`결재해`처럼 승인 의도가 명확한 형태는 §5.2F의 승인 자연어 명령이 우선 가로챈다.)
* **트리거 동사**: 핵심 명사 뒤에 `올려줘`/`해줘`/`해주`/`써줘`/`써주`/`하겠다`/`할래`/`할게`/`부탁`/`상신`/`등록` 등의 동사가 붙는 경우는 물론, 탭/줄바꿈으로 구분된 항목 뒤에 별도 동사 없이 핵심 명사로 문장이 끝나는 경우(예: `"2026/01/15 21:54 본죽&비빔밥 여의도점 61000 품의"`, 또는 `"결재"`/`"상신"` 단독 입력)도 동일하게 품의 등록 요청으로 인식한다.
* **모호한 입력에 대한 확인 질문**: 핵심 명사는 포함되어 있으나 트리거 동사도 없고 문장 끝에 오지도 않아 구체적인 흐름으로 특정할 수 없는 경우(예: `"품의 어떻게 하는거야?"`), 임의로 추측해 실행하지 않고 `"품의 작성을 도와드릴까요?"`처럼 확인 질문을 되묻는다.
* **날짜 파싱 지원**: 하이픈(`-`), 슬래시(`/`), 닷(`.`), 한글(`월 일`) 형태의 날짜 포맷을 모두 정규화하여 인식한다.
* **상호명 퍼지 매칭(Fuzzy Matching)**: 사용자의 오탈자를 고려하여 Levenshtein 거리 알고리즘 기반 매칭을 적용한다 (예: "파리바게트" 입력 시 "파리바게뜨" 검색 허용).
* **예외 처리**:
  * 조건 만족 조건이 복수 개(중복 후보)일 경우: 무작위 선택을 금지하고, 후보 목록을 사용자에게 제시하며 구체적인 재입력을 요청한다.
  * 조건 없이 `"품의하겠다"`/`"결제건 품의해줘"`처럼 날짜·시간·상호명·금액 조건이 전혀 없이 요청할 경우: 무작위로 고르지 않고, 현재 부서의 미사용 카드결제 후보 목록(최대 10건)을 번호와 함께 제시한다. 사용자가 번호(예: `"1"`, `"1번"`)로 응답하면 해당 건으로 AI 분석·적요 추천 단계(§5.2A 상단)로 진행하고, 이어서 참석자/내용(사유) 입력을 유도한다 — 이 선택 대기 상태(`AWAITING_CARD_SELECTION`)에서 번호 범위를 벗어나거나 숫자가 없는 응답에는 다시 번호로 골라달라고 재요청한다.

#### B. 적요 변경 자연어 명령

* AI가 추천한 계정과목(적요)에 대해 사용자가 `"회의비로 바꿔줘"`, `"교통비로 변경"` 등 명시적으로 요청하면, 실시간으로 계정과목을 변경 적용한다 (`tryHandleAccountChange`).
* 이는 참석자 입력 단계 전/후 상관없이 상시 작동해야 하며, 계정 변경 시 기입력된 정보를 바탕으로 아래의 사유 문구를 자동 재조립한다.

#### C. 사유 문구(적요 내용) 자동 생성 템플릿

`AWAITING_ATTENDEES` 단계에서는 일반 채팅 입력창 대신 사이드바에 **"참석자 :" / "사유 :" 전용 입력칸 2개 + 제출 버튼**을 별도로 표시한다 (일반 채팅 입력창은 그대로 아래에 남아있어 "회의비로 바꿔줘" 같은 적요 변경 명령은 이 단계에서도 계속 자유 텍스트로 처리된다). 제출 시 두 칸의 값을 `"참석자 : {값}\n사유 : {값}"` 형태로 합쳐 기존 채팅 파이프라인에 그대로 전달하고, 입력받은 문자열을 파싱(`parse_attendee_input`)하여 계정과목별로 아래 템플릿에 맞추어 사유 문구를 자동 생성한다. 파싱은 `참석자 : .../ 사유(또는 내용) : ...` 라벨 형식(한 줄이든 여러 줄이든)을 최우선으로 인식하고, 라벨이 없으면 기존의 슬래시(`/`) 구분 형식으로, 그마저 없으면 슬래시가 없는 일반 공백 형태의 문장에서 마지막 단어/내용을 구분하는 폴백 로직을 순서대로 시도한다.

* **본사 회의비**: `"[최초참석자] 외 N인 [내용] 관련회의"`
* **본사 조직활성화비**: `"[최초참석자] 외 N인 [내용]관련 업무활성화로 인한 조직활성화비"`
* **본사 업무추진비**: `"[최초참석자] 외 N인 [내용] 관련 업무추진비"`
* *특이사항*: 예산 부족으로 1건이 2개 계정(예: 회의비 + 업무추진비)으로 분할될 경우, 분할 생성된 각 건의 적요 라인은 **자신의 계정 템플릿에 맞는 사유 문구를 각각 독립적으로 생성**해야 한다.

#### D. 예산 잔액 조회 명령 (`isBudgetQuery`)

* 사용자가 `"예산 얼마 남았어?"`, `"전화료 남았어?"` 등 예산 잔액을 질의하는 자연어를 인식한다.
* 특정 적요(예: 전화료)가 언급된 경우 해당 적요의 예산 정보만 핀포인트로 답변하고, 적요 언급이 없을 경우 5종 적요 전체의 배정/사용/잔여 현황 요약을 출력한다.

#### E. 최종 확인 대화 흐름

* 최종 품의 상신 전 단계("이대로 최종 품의 상신을 올릴까요?")에서 시스템은 다음 긍정어 목록을 완벽히 인식하여 최종 처리를 수행해야 한다.
* *인식 대상*: `응`, `웅`, `네`, `넵`, `넹`, `확인`, `어`, `오케이`, `ok`, `okay`, `yes`, `ㅇㅇ`, `ㅇ케이`
* 이 자유 텍스트 인식과 별개로, `CONFIRMATION`/`SHORTFALL_CONFIRMATION`/`APPROVAL_CONFIRMATION` 세 확인 대기 상태 공통으로 채팅창에 **"✅ 예" / "❌ 아니오" 버튼**을 함께 제공한다 — 버튼은 각각 `"응"`/`"아니오"` 텍스트를 보낸 것과 동일하게 처리되어 위 긍정어 인식 로직을 그대로 재사용한다.
* **회의록 PDF 업로드 강제 (§3.3 증빙 서류 필수 한도)**: `CONFIRMATION` 단계에서 현재 추천/변경된 적요가 본사 회의비이고 금액이 10만원 이상이면(`recommended_proposal.requiresMinutes`), 챗봇 화면에도 [1. 지출결의 입력] 탭의 수동 폼과 동일하게 회의록 PDF 업로드칸을 노출한다. 파일이 첨부되지 않은 채로 긍정 응답("응"/"✅ 예" 등)을 보내면 상신을 진행하지 않고 업로드를 먼저 요청하는 안내를 되돌려준다(확인 대기 상태는 그대로 유지). 새로운 품의 흐름이 시작될 때마다 이전 건에서 첨부했던 파일은 초기화되어, 서로 다른 건에 같은 PDF가 잘못 재사용되지 않는다.

#### F. 지출품의 승인 및 취소(반려) 자연어 명령

챗봇을 통해 '지출품의승인(부서장)' 탭과 동일한 승인/승인취소(반려) 처리를 수행할 수 있다. 처리 대상, 검증 규칙, 부수 효과(사용됨 마킹 해제 등)는 §4.1·§4.2의 기존 규칙을 그대로 따르며, **처리 대상은 미승인(결재대기) 상태의 건으로 한정**한다 — 이미 승인 완료된 건은 챗봇을 통한 취소(반려) 대상이 되지 않으며, 해당 건이 지목된 경우 이미 승인 완료되어 처리할 수 없다는 사실을 안내한다.

* **의도 인식**:
  * 승인 의도 키워드: `승인해줘`, `승인해`, `결재해줘`, `승인 처리` 등
  * 취소(반려) 의도 키워드: `승인취소`, `취소해줘`, `반려해줘`, `반려` 등
  * 한 문장에 두 의도 키워드가 모두 감지되는 경우 취소(반려) 의도를 우선 적용한다.

* **대상 지정 방식**: 아래 두 가지 방식을 모두 지원한다.
  1. **품의번호(ID) 지정**: 품의 상신 완료 시 안내된 결재대기 번호를 직접 언급한다 (예: `"1001번 승인해줘"`, `"1001, 1002 취소해줘"`). 콤마 또는 공백으로 구분된 다중 ID 입력을 지원한다.
  2. **조건 검색 지정**: 부서명, 금액, 적요(계정과목명), 내용(사유 문구 일부) 등 조건으로 미승인 건을 검색한다 (예: `"스타벅스 건 승인해줘"`, `"IT정보팀 미승인 건 취소해줘"`). 내용/상호 조건 매칭은 카드결제 검색(§5.2A)과 동일한 퍼지 매칭(Levenshtein 기반)을 재사용한다.

* **전체/일괄 처리**: `전체`, `모두`, `다`, `일괄` 등의 키워드가 포함된 경우, 조건에 해당하는 **모든 미승인 건**을 한 번에 승인 또는 취소 대상으로 삼는다 (예: `"미승인 건 전부 승인해줘"`, `"IT정보팀 건 전체 취소해줘"`). 별도 조건 없이 `"전체 승인해줘"`라고만 요청하면 현재 존재하는 모든 미승인 건이 대상이 된다.

* **모호성(중복 후보) 처리**: 일괄 처리 키워드나 품의번호(ID) 지정 없이 대상이 2건 이상인 경우, 임의로 전체를 처리하지 않고 **번호가 매겨진 후보 목록**(예: `"1. [1001] IT정보팀 · 회의비 · 88,000원"`, 품의번호/부서/적요/금액 포함, 최대 8건 표시)을 제시하며 `AWAITING_APPROVAL_SELECTION` 대기 상태로 전환한다(카드결제 검색의 기존 ambiguous 처리와 동일한 원칙 — 무작위 선택 금지). **조건 검색 결과가 2건 이상인 경우뿐 아니라, `"승인해줘"`/`"취소해줘"`처럼 아무 조건도 지정하지 않아 현재 미승인 건 전체가 대상이 되는 경우도 동일하게 이 목록 제시 흐름을 탄다** (§5.2A에서 조건 없는 `"품의하겠다"` 요청 시 후보 목록을 보여주는 것과 같은 원칙).
* **번호로 선택**: `AWAITING_APPROVAL_SELECTION` 상태에서 사용자가 목록의 번호(예: `"1"`, `"1, 3"`처럼 콤마/공백으로 구분한 다중 선택도 지원)로 응답하면, 그 번호에 해당하는 건(들)만 골라 최종 확인 단계(§5.2E)로 진행한다. 물론 품의번호를 직접 지정(`"1001번 승인해줘"`)하거나 `전체`/`모두` 등을 포함해 재요청하는 기존 방식도 그대로 유효하다. 목록에 없는 번호나 숫자가 아예 없는 응답에는 다시 번호로 골라달라고 재요청한다.

* **최종 확인 절차**: 대상이 특정된 후, 실제 승인/취소 실행 전 반드시 다음 정보를 요약해 제시하고 사용자의 긍정 응답(§5.2E와 동일한 긍정어 목록)을 받은 뒤에만 처리를 수행한다.
  * 대상 건수, 각 건의 품의번호·부서·적요·금액, 대상이 2건 이상이면 합계 금액도 함께 표시
  * 예: `"다음 2건을 승인 처리할까요? [1001] IT정보팀 · 회의비 · 88,000원 / [1002] IT정보팀 · 업무추진비 · 30,000원 (합계 118,000원)"`
  * 확인 대기 중에는 신규 품의 작성 등 다른 챗봇 플로우로 전환되지 않으며, 부정 응답 또는 무관한 입력이 들어오면 처리를 취소하고 대기 상태(IDLE)로 되돌린다.

* **실행 및 부수 효과**:
  * 승인 실행 시 대상 건(들)의 상태를 '승인'으로 변경한다 (§4.1과 동일).
  * 취소(반려) 실행 시 대상 건(들)을 품의 데이터에서 삭제하며, §4.2의 복원 규칙(예산 분할로 파생된 모든 형제 라인이 함께 취소되어야만 해당 카드결제의 '사용됨' 마킹이 해제됨)을 동일하게 적용한다.
  * 처리 완료 후 처리된 건수와 품의번호 목록을 챗봇 메시지로 안내한다.

## 6. 시스템 아키텍처 및 범위 결정 (Architecture Decisions)

j.html 프로토타입을 Next.js로 이관하되, **범위는 프로토타입 수준을 유지**하기로 결정한다. 아래 항목은 실서비스 전환 시 재검토가 필요한 TBD로 명시해둔다.

### 6.1 데이터 저장소

* **결정**: 별도 백엔드 DB 없이 **브라우저 `localStorage`를 유지**한다. j.html과 동일하게 `dataset`, `cardUsageCache`, `usedCardUsageKeys`를 클라이언트에 영속화한다.
* Next.js 구현 시에도 서버 API/DB 계층을 새로 만들지 않고, 클라이언트 컴포넌트에서 `localStorage`를 직접 다룬다 (SSR 시 hydration mismatch 방지를 위해 초기 렌더는 빈 상태 → `useEffect`에서 로드).
* **TBD**: 실운영 전환 시 다중 사용자/기기 간 데이터 공유가 필요해지면 실제 DB(PostgreSQL 등)로 교체 필요.

### 6.2 사용자 인증 및 권한

* **결정**: 별도 로그인 시스템 없이, 화면 상단 등에 **역할 전환 버튼(일반사용자 ↔ 부서장)**을 두어 뷰/탭 노출을 제어한다 (j.html 방식 유지).
* '지출품의승인(부서장)' 탭은 부서장 역할일 때만 노출/접근 가능하다.
* **TBD**: 실운영 전환 시 실제 로그인 및 서버 사이드 권한 검증(RBAC) 필요. 현재는 클라이언트 단 역할 전환이므로 보안 경계가 없음을 인지하고 프로토타입 용도로만 사용한다.

### 6.3 챗봇 NLP 엔진 아키텍처

* **결정**: **하이브리드 방식** — 규칙 기반 파서(정규식 날짜/시간/금액 추출, Levenshtein 퍼지 매칭, `isBudgetQuery`/`tryHandleAccountChange` 등 기존 로직)를 1차로 시도하고, 규칙이 의도를 확정하지 못한 경우에만 LLM API(Claude)로 폴백하여 의도 파악 및 응답을 생성한다.
* 폴백 트리거 조건(예: 규칙 매칭 실패, 후보 0건, 애매한 자유발화 등)과 LLM 호출 시 전달할 컨텍스트(현재 dataset 상태, 예산 현황, 카드 후보 목록 등)는 구현 단계에서 상세 설계한다.
* 정합성이 중요한 최종 확정 로직(예산 분할 계산, 사유 문구 템플릿 조립, 승인취소 복원 규칙)은 **항상 규칙 기반 코드로만 처리**하며 LLM 출력에 의존하지 않는다.

### 6.4 부서/카드 마스터 데이터 범위

* **결정**: 현재 범위는 **C30 IT정보팀 단일 부서**만 지원한다. 부서 선택 UI는 두더라도 실질적으로는 C30 고정이며, 카드 목록도 C30 소속 카드로 한정한다.
* **TBD**: 여러 부서/카드 지원은 확장 단계 과제로 남긴다 (부서별 예산 분리, 부서 선택에 따른 카드 목록 필터링 등).

### 6.5 데이터 모델 (초안)

구현 참고용으로 `localStorage`에 저장되는 핵심 엔티티의 최소 필드를 정의한다. 실제 구현 중 세부 필드는 조정 가능하다.

* **`dataset` (품의 라인)**: `id`, `department`(C30 고정), `accountCode`(08WA/40AA/45BC/12CC/20DD), `amount`, `reasonText`(자동생성 사유 문구), `docDate`(상신일), `status`(대기/승인/반려), `sourceUsageKeys`(string[], 원본 카드결제 키 참조), `attachmentRequired`(boolean), `attachmentFile`(회의록 PDF 등)
* **`cardUsageCache` (카드결제 내역)**: `usageKey`(고유키), `cardNumber`, `cardOwner`, `merchant`, `amount`, `paidAt`(일시), `location`
* **`usedCardUsageKeys`**: `usageKey[]` — 이미 품의로 소진된 카드결제 키 목록
* **예산 테이블**: `accountCode`별 `allocatedBudget`, `usedBudget`, `remainingBudget` (5종 계정과목 고정 레코드)

## 8. UI/UX 디자인 컨셉 (라이트 테마 리디자인)

`app.py`(Streamlit 포팅판)의 화면을 더 깔끔하고 전문적인 인상으로 개선하기 위한 디자인 명세. **화면/레이아웃 구조 자체(좌측 사이드바 챗봇 패널 + 메인 영역 3개 탭)는 그대로 유지**하며, 컬러 테마·타이포그래피·컴포넌트 스타일만 개선하는 범위로 한정한다.

### 8.1 컬러 테마

* **베이스**: 라이트 모드 (Streamlit `theme.base = "light"`) — 다크 테마는 너무 어둡다는 피드백에 따라 라이트 모드로 전환했다.
* **하이라이트(강조) 색상**: 인디고 보라 계열 (`primaryColor = #6366F1`) — 기존 챗봇 헤더(인디고)와 톤을 통일한다. 주요 액션 버튼(품의 등록, 일괄 승인 등 `type="primary"` 버튼), 챗봇 상태 배지, 카드형 컨테이너 강조 테두리에 일관되게 적용한다.
* **경고/에러 색상**: Streamlit 기본 팔레트(빨강/주황 계열)를 그대로 사용해 예산초과·검증 실패 등 기존 경고 시맨틱을 라이트 테마에서도 그대로 유지한다.

### 8.2 타이포그래피

* 가독성 위주의 산세리프 폰트 스택 (`'Pretendard', -apple-system, BlinkMacSystemFont, 'Segoe UI', system-ui, sans-serif`) — j.html 프로토타입에서 사용하던 폰트 스택과 동일하게 맞춘다.
* 별도 외부 폰트 파일(CDN)은 로드하지 않는다 — 오프라인/사내망 환경에서도 동일하게 동작해야 하므로 시스템 폰트 폴백에 의존한다.

### 8.3 컴포넌트 배치 및 카드형 그룹핑

* 레이아웃 구조는 변경하지 않는다: 메인 영역은 여전히 3개 탭(지출결의 입력 / 지출품의 승인 / 승인현황·예산내역), 챗봇은 여전히 좌측 사이드바 고정 패널.
* 정보 밀도가 높은 블록(카드결제 사용내역 조회+추천, 회계 분개 입력, 승인 대기 목록, 승인현황/예산 리포트 테이블)은 각각 **테두리가 있는 카드형 컨테이너**(`st.container(border=True)`)로 감싸 시각적으로 구분한다.
* 챗봇 안내 문구(`st.info`/`st.warning` 등)는 기존처럼 인라인으로 두되, 이를 감싸는 상위 섹션이 카드 컨테이너로 구분되어 있어 전체적으로 그룹핑된 느낌을 준다.
* 사이드바 최상단 브랜딩 타이틀("📊 ERP 지출관리 시스템")은 세로 공간을 아끼기 위해 작은 폰트의 컴팩트 표기로 축소하고, 대신 AI 에이전트 영역에 더 넓은 시각적 비중을 준다.
* 사이드바 내부 순서는 **① 브랜딩/부서·로그인 정보 → ② AI 에이전트 배너+상태배지 → ③ 챗봇 대화 이력 → ④ 챗봇 입력창 → ⑤ 개발/테스트 전용 도구("🔧 테스트 도구": 가상 카드결제 발생, 테스트 데이터 초기화)** 순으로 배치한다. 실제 업무 흐름(챗봇 대화)과 개발용 유틸리티를 시각적으로 분리하기 위해 테스트 도구는 항상 챗봇 입력창보다 아래에 위치시킨다.

### 8.4 AI 에이전트(챗봇) 강조

* 사이드바 챗봇 영역 상단에 **AI 에이전트 배너 + 상태 배지 + 애니메이션 점멸 인디케이터**를 배치해 AI 에이전트가 항상 활성 상태임을 시각적으로 강조한다.
* **AI 에이전트 배너**: "AI 지출결의 에이전트" 타이틀을 인디고→보라 그라데이션 배경의 배너로 감싸고, 일반 텍스트보다 큰 글씨(약 1.2rem)로 키운다. 다만 글자 굵기는 **볼드를 적용하지 않은 일반체**로 두어 큰 그라데이션 배경 위에서 과하게 무거워 보이지 않도록 한다.
* 배너 아이콘은 🤖(로봇) 대신 **💡✨** 조합을 사용한다 — 로봇 이모지는 인디고/보라 그라데이션 배경과 명도 대비가 약해 잘 보이지 않는 문제가 있어 교체했다 (챗봇 메시지 아바타 등 다른 위치의 🤖는 그대로 유지).
* 배지는 현재 대화 상태(`chatbot_step`)를 사람이 읽을 수 있는 라벨로 매핑해 보여준다: 대기 중 / 참석자 정보 입력 대기 / 최종 확인 대기 / 예산분할 확인 대기 / 승인·취소 확인 대기.
* 인디케이터 점은 j.html 원본의 초록색 `animate-ping` 효과와 동일한 느낌으로 CSS `@keyframes`를 이용해 은은하게 점멸시킨다.

### 8.5 구현 방식

* 라이트 테마 베이스/강조색은 `.streamlit/config.toml`의 `[theme]` 설정으로 지정한다.
* 폰트·버튼·카드·배지 등 세부 스타일은 앱 최상단에서 `st.markdown(<style>..., unsafe_allow_html=True)`로 한 번 주입하는 커스텀 CSS로 처리한다.
* 카드형 그룹핑은 커스텀 HTML 대신 Streamlit 내장 `st.container(border=True)`를 우선 사용해, 유지보수 부담과 Streamlit 버전 호환성 리스크를 낮춘다. 배지/점멸 인디케이터처럼 Streamlit 기본 컴포넌트로 표현할 수 없는 순수 장식 요소만 커스텀 CSS/HTML로 구현한다.
