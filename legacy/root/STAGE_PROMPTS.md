# Automate-X 모노레포 전환 — 단계별 실행 프롬프트

PRD 원문과 기존 3개 모듈(`AutoX/manual`, `AutoX/nl2sql`, `AutoX/rpa`)의 실제 코드를 확인한 뒤 작성한 실행용 프롬프트 모음.
각 단계는 이전 단계의 결과물을 전제로 하므로, 순서대로 새 대화(또는 이어지는 대화)에 그대로 붙여넣어 사용.

---

## 0. 실행 전 확인 (1단계 프롬프트에 포함되어 있지만 별도로도 인지할 것)

- `AutoX/.env` (GEMINI_API_KEY), `AutoX/manual/.env` (OPENAI_API_KEY) — 평문 키 노출. 모노레포화 전에 로테이션 + `.gitignore` 처리 필요.
- `AutoX/nl2sql`은 `origin=github.com/rkdals0203/im-one.git`을 가진 독립 git 저장소. 히스토리 보존 여부를 먼저 결정해야 함(git subtree 흡수 / submodule 유지 / 히스토리 폐기 후 단순 복사).
- `manual`(Flask), `nl2sql`(LangGraph 패키지 + 순수 http.server), `rpa`(Streamlit)는 프레임워크가 전부 다름 — "하나의 인터페이스"라는 PRD 목표와 지금의 파편화된 실행 방식 사이에 간극이 있음.

---

## 1단계: 기본 뼈대 작성 (모노레포 스캐폴딩)

```
Automate-X를 모노레포로 재구성한다. 현재 c:\Users\moon\AutoX 아래에
manual/(모듈1: AI 지식정보 저장소), nl2sql/(모듈2: NL2SQL), rpa/(모듈3: 행정 자동화)
세 폴더가 각각 독립적으로 존재한다. PRD.md(AutoX/rpa/PRD.md에 있는 것과 별개로,
방금 논의한 상위 PRD 전체)를 기준으로 아래를 수행해줘.

1. 구조 설계
   - apps/manual, apps/nl2sql, apps/rpa 로 3개 앱을 이동 (또는 동등한 워크스페이스 구조 제안).
   - 루트에 PRD.md, README.md(모노레포 전체 설명), .gitignore, .env.example을 둔다.
   - nl2sql은 자체 git 저장소(origin: github.com/rkdals0203/im-one.git)를 갖고 있다.
     git subtree로 흡수할지, 히스토리를 버리고 단순 복사할지 결정해야 하니
     작업 전에 반드시 나에게 확인해라 (임의로 git history를 지우지 말 것).
   - 세 앱이 서로 다른 스택(Flask / 순수 http.server+LangGraph / Streamlit)을 쓰고 있어
     의존성을 강제 통합하지 말고, 앱별 격리(각자 venv 또는 uv workspace의 독립 member)를
     유지하면서 루트에서 공통으로 관리할 것(공통 lint/format 설정, 공통 스크립트 등)만 통합해라.

2. 보안/환경변수 정리
   - AutoX/.env(GEMINI_API_KEY), AutoX/manual/.env(OPENAI_API_KEY)가 평문으로 존재한다.
     이런 파일이 새 저장소에 커밋되지 않도록 .gitignore에 명시하고,
     각 앱 폴더에 .env.example만 남기고 실제 값이 든 .env는 커밋 대상에서 빼라.
   - 이미 노출된 키는 재발급(로테이션)이 필요하다는 것을 나에게 경고해라. 네가 직접
     키를 재발급하거나 외부에 전송하지는 마라.

3. 실행 검증
   - 이동 후에도 세 앱이 각자 기존 방식(python app.py / streamlit run app.py / im-one-agent-web 등)으로
     동일하게 기동되는지 확인해라.
   - 각 앱의 CLAUDE.md(manual/CLAUDE.md, rpa/CLAUDE.md)에 적힌 실행/제약 사항을 위반하지 않았는지 확인해라.

4. 완료 기준
   - 루트 README에 모노레포 구조, 각 앱 실행 방법, PRD 모듈-폴더 매핑표를 작성.
   - 아직 손대지 않은 "하나의 인터페이스로 통합"은 3단계(UI) 과제로 남겨두고, 지금은
     구조적 스캐폴딩과 안전한 이동/정리에만 집중해라.
```

**왜 이렇게 짰는지**
- 세 앱의 스택이 이미 크게 다르므로("뼈대"라고 의존성을 억지로 합치면 오히려 깨짐), 이 단계는 *물리적 재배치 + 보안 정리 + 실행 검증*에 한정.
- git 히스토리/키 로테이션처럼 되돌리기 어려운 작업은 에이전트가 임의로 처리하지 않고 먼저 물어보도록 명시.

## 2단계: 기능 (PRD 대비 기능 갭 구현)

```
1단계에서 재구성한 모노레포(apps/manual, apps/nl2sql, apps/rpa)를 대상으로,
PRD 3.1~3.3의 각 워크플로우 요구사항과 실제 코드 사이의 갭을 찾아 구현해라.
먼저 갭 분석 결과를 표로 보여주고, 내 확인을 받은 뒤 구현에 들어가라.

[모듈1: apps/manual — PRD 3.1]
- 현재 bond_manual.md(채권 매뉴얼)만 질의응답이 서빙되고 있다. meeting_room_manual.md(회의실예약 매뉴얼)도
  같은 방식으로 질문 라우팅되는지 확인하고, 안 되면 두 소스를 함께 검색하도록 통합해라.
- PRD는 "LLM 정제"와 "출처 기반 정확한 응답"을 요구하지만 현재는 키워드 검색만 동작하고
  OpenAI 키가 401로 막혀 있다(manual/CLAUDE.md 참고). LLM 연동을 강제로 진행하지 말고,
  키 문제를 먼저 나에게 확인해라. 연동 전까지는 지금의 키워드 검색 품질(TOP_K, IDF 가중치)을
  개선하는 방향으로 진행해도 좋다.
- PRD 3.1의 "데이터 입력 및 파싱"(동영상/메모/규정집 업로드)이 현재는 사전 처리된 정적
  md 파일 서빙뿐이다. 실제 업로드→파싱 파이프라인이 필요한지, 지금 PoC 범위에서는
  생략해도 되는지 나에게 물어봐라.

[모듈2: apps/nl2sql — PRD 3.2 Step1~7]
- graph.py/intent_guard.py/schema_retrieval.py/sql_generator.py/sql_safety.py/
  database_backend.py/response.py가 각각 PRD의 Step2~7에 대응하는지 매핑표를 만들어라.
- evaluate.py/evaluation.py를 이용해 PRD 핵심지표("자연어 질의 SQL 실행 정확도 80% 이상")를
  실제로 측정할 수 있는 재현 가능한 명령(예: im-one-agent-evaluate)이 있는지 확인하고,
  없다면 만들어라. 결과를 리포트 형태로 남겨라.
- PRD 5장 리스크 대응책인 "Self-healing 프롬프트 루프"가 sql_safety.py의 검증/보정 단계에
  실제로 구현되어 있는지 확인해라.

[모듈3: apps/rpa — PRD 3.3]
- 현재 app.py는 이벤트 기반(가상 카드결제 감지, 챗봇 명령)만 지원한다. PRD가 요구하는
  "매월 25일 오전 10시" 같은 시간 트리거 예약 기능이 있는지 확인하고 없다면 설계를
  제안해라(스케줄 실행 방식은 Streamlit 특성상 별도 백그라운드 프로세스가 필요할 수 있으니
  구현 전에 방식을 나와 논의해라).
  - 이미 rpa/PRD.md와 rpa/app.py 사이에 과거 불일치가 한 번 발견된 적이 있다
    (회의비/조직활성화비 PDF 증빙 규칙, rpa/CLAUDE.md 참고). 새 기능을 추가할 때도
    PRD.md와 app.py를 항상 동기화해라. 둘이 다르면 어느 쪽이 맞는지 나에게 먼저 물어보고
    임의로 한쪽을 "고치지" 마라.
  - "작업 완료 알림"(결과물/초안 링크를 플랫폼 알림으로 보고) 기능이 있는지 확인해라.

각 모듈 구현이 끝나면 해당 모듈의 CLAUDE.md/PRD.md를 갱신해서 실제 구현과 문서가
어긋나지 않게 해라.
```

**왜 이렇게 짰는지**
- 세 모듈 다 "이미 상당 부분 구현되어 있다"는 걸 코드로 확인했기 때문에, 처음부터 새로 만들라는 프롬프트 대신 *갭 분석 → 확인 → 구현* 순서로 짜서 중복 작업을 방지.
- rpa 모듈은 PRD.md와 코드가 어긋난 전례가 실제로 있어서, 이번에도 같은 실수가 나지 않게 명시적으로 못 박음.

## 3단계: UI 디자인 (통합 인터페이스)

```
PRD의 핵심 개념은 "인력 공백 방지, 데이터 병목 해소, 반복 행정 업무 자동화를
하나의 인터페이스(채팅/대시보드)에서 해결"하는 것이다. 하지만 현재 apps/manual(Flask+Jinja),
apps/nl2sql(순수 http.server 정적 페이지), apps/rpa(Streamlit)는 서로 다른 프레임워크로
완전히 분리되어 있고 디자인도 통일되어 있지 않다.

먼저 통합 방식에 대한 옵션을 정리해서 나에게 제시하고, 내가 하나를 고른 뒤에 작업을 시작해라
(이 결정은 이후 유지보수 방식을 크게 바꾸므로 임의로 진행하지 마라):

  A) 각 앱은 그대로 두고, 공통 랜딩/게이트웨이 페이지에서 세 앱으로 라우팅
     (디자인 시스템만 공유: 색상/타이포/레이아웃 토큰 통일)
  B) 세 앱을 하나의 Streamlit 멀티페이지 앱으로 재구성
     (nl2sql/manual의 백엔드 로직은 그대로 두고 프레젠테이션 레이어만 Streamlit으로 이식)
  C) 별도 프론트엔드(예: Next.js)를 새로 만들고 세 앱을 API 서버로 뒤에 둔다

결정된 방식에 따라 아래를 진행해라:

1. 디자인 토큰 통일
   - apps/rpa/app.py에 이미 정의된 인디고(#6366F1) 액센트 + CUSTOM_CSS(rpa/CLAUDE.md
     "app.py styling" 절 참고)를 기준 디자인 시스템으로 삼을지 검토하고, manual/nl2sql
     UI에도 같은 색상/타이포 톤을 적용해라.
2. 공통 UX 요소
   - 채팅 인터페이스(모듈1 질의응답, 모듈2 NL2SQL 질의, 모듈3 챗봇)를 시각적으로
     동일한 컴포넌트처럼 보이게 통일해라.
   - 모듈2의 결과 시각화(PRD 3.2 Step7: pandas 기반 차트/테이블)가 실제로 붙어있는지
     확인하고, 없다면 추가해라.
3. 접근성/반응형
   - 라이트/다크 모드, 좁은 화면 대응 여부를 확인해라.

작업 전 각 모듈의 CLAUDE.md에 적힌 UI 관련 제약(예: rpa의 st.chat_input 사이드바 배치
특성, Streamlit rerun 모델 제약)을 반드시 확인하고 위반하지 마라.
```

**왜 이렇게 짰는지**
- "하나의 인터페이스"라는 PRD 목표와 현재의 3-프레임워크 파편화 사이 간극이 가장 큰 리스크라서, 이 단계는 구현보다 먼저 **통합 방식 결정**을 요구하도록 설계 (되돌리기 어려운 아키텍처 선택이라 사용자 승인 필요).
- rpa에 이미 존재하는 디자인 토큰을 기준점으로 재사용하도록 유도해 중복 디자인 작업 방지.

## 4단계: PRD 적합성 및 버그 검토

```
지금까지 구현된 apps/manual, apps/nl2sql, apps/rpa 전체를 PRD 원문 기준으로 상세히 검토해라.

1. PRD 체크리스트 (충족 / 부분충족 / 미충족으로 분류, 근거 파일:라인 명시)
   - 2.1 지식의 시스템화: 온보딩 기간 단축을 측정할 방법이 실제로 있는가, 그냥 목표로만
     남아있는가?
   - 2.2 데이터 접근 민주화: nl2sql의 evaluate.py 결과 기준 "자연어 질의 → SQL 실행 정확도
     80% 이상"이 실제로 달성/측정되는가?
   - 2.3 고부가가치 업무 전환: rpa 모듈에서 지출품의 처리 시간이 실제로 측정 가능한가
     (2분 이내 목표), 아니면 정성적 주장뿐인가?
   - 3.1~3.3 각 워크플로우 단계가 실제 코드의 어느 함수/파일에 대응하는지 매핑하고,
     빠진 단계가 있으면 명시해라.
   - 4.2 보안 지침: 개인정보/실제 금융사 비공개 데이터가 코드/데이터 파일에 섞여 있지
     않은지 확인해라 (manual/CLAUDE.md에 이미 "개인명 pseudonymize" 규칙이 있으니
     그 규칙이 실제로 지켜졌는지 재검증해라).
   - 5장 리스크 대응책(Self-healing SQL 루프, 노이즈 데이터 20% 이상 혼합 테스트)이
     실제로 구현/테스트되어 있는지 확인해라.

2. 버그 및 불일치 탐지
   - 각 모듈의 PRD.md/CLAUDE.md와 실제 코드 동작이 어긋나는 곳을 찾아라
     (rpa는 과거에 실제로 한 번 발견된 전례가 있다 — 같은 유형의 새로운 불일치가
     또 생겼는지 확인).
   - 재현 조건(입력값, 실행 경로)을 구체적으로 적어서 보고해라. 발견한 문제를
     네가 임의로 "이게 맞겠지"하고 고치지 말고, 어느 쪽(문서 vs 코드)이 의도인지
     먼저 나에게 확인해라.
   - 노출된 API 키(AutoX/.env, apps/manual/.env)가 로테이션되었는지, .gitignore로
     보호되고 있는지 다시 확인해라.

3. 출력 형식
   - PRD 항목별 체크리스트 표
   - 발견된 버그/불일치 목록 (파일:라인, 재현 조건, 심각도)
   - 위 두 가지를 종합한 "다음에 우선 처리해야 할 항목" 우선순위 리스트
```

**왜 이렇게 짰는지**
- PRD의 정량 지표(50% 단축, 80% 정확도, 85% 절감)를 "측정 가능한가"로 검증하도록 강제 — 정성적 주장만 남는 걸 방지.
- rpa에서 실제로 있었던 문서-코드 불일치 전례를 인용해, 같은 클래스의 버그를 다시 찾도록 유도.
- 발견한 문제를 에이전트가 마음대로 고치지 않고 먼저 확인받게 해서, 의도된 동작을 실수로 "버그"라 판단해 망가뜨리는 걸 방지.
